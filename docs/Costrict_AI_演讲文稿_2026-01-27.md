## 幻灯片 2: 核心问题

### 标题
**AI 辅助开发的三大痛点**

### 描述
**痛点 1：AI 会"失忆"**
- ❌ 忘记你用的技术栈
- ❌ 忘记你实现某个功能的方式
- ❌ 忘记你踩过的坑和解决方案

**痛点 2：上下文过长导致幻觉**
- ❌ 对话太长，AI 开始胡说八道
- ❌ 重复犯同样的错误
- ❌ 生成的代码前后矛盾

**痛点 3：AI 犯错后深陷泥潭**
- ❌ 一个错误引发连锁反应
- ❌ 改来改去越改越乱
- ❌ 不知道如何回滚

### 演讲稿
> "大家好，今天我想分享我用 Costrict 开发 GlobalInSight 项目的实战经验。
> 
> GlobalInSight 是一个复杂的全栈项目：后端用 FastAPI + LangGraph 构建多 Agent 系统，前端用 Vue 3 全家桶，整个系统包含热点采集、LLM 解读、小红书发布等功能。
> 
> 在开发过程中，我遇到了 AI 辅助开发的三大痛点：
> 
> **第一，AI 会失忆**。虽然它不会忘记 FastAPI 是什么，但会忘记你是怎么实现某个功能的，会忘记你踩过的坑。比如我告诉它'用 LangGraph 的 StateGraph'，下次它可能又建议用普通的 Graph。
> 
> **第二，上下文过长导致幻觉**。当对话超过 50 条消息，AI 开始胡说八道。它会重复犯同样的错误，生成的代码前后矛盾。特别是开发大组件时，这个问题很严重。
> 
> **第三，AI 犯错后深陷泥潭**。一个小错误引发连锁反应，改来改去越改越乱。你不知道该回滚到哪个版本，只能推倒重来。
> 
> 针对这些问题，我总结了 6 个实战技巧。这些技巧让我的开发效率提升了 40%，代码质量反而更高。"

## 幻灯片 3: 技巧 1 - 建立文档库让 AI 学习

### 标题
**解决 AI 失忆：建立项目知识库**

### 描述
**问题**：
- AI 忘记你实现某个功能的方式
- AI 忘记你踩过的坑和解决方案
- 每次都要重新解释项目背景

**我的做法**：
```
docs/
├── README.md                    # 文档导航中心
├── 文档格式规范.md               # 文档编写格式
├── API接口文档.md                # 前后端共享的API规范
├── 产品需求总览.md               # 项目总体需求
├── 前端/
│   ├── 需求清单/
│   ├── 已实现功能/              # ⭐ 记录实现方式
│   ├── 技术文档/
│   ├── 测试文档/
│   └── 问题排查/                # ⭐ 记录踩坑经验
├── 后端/
│   ├── 需求清单/
│   ├── 已实现功能/
│   ├── 技术文档/
│   ├── 测试文档/
│   └── 问题排查/
└── 部署运维/
```

**关键原则**：
- ✅ 每次实现功能后，立即记录到"已实现功能"
- ✅ 每次踩坑后，立即记录到"问题排查"
- ✅ 文档越多，AI 越聪明

### 演讲稿
> "第一个技巧是建立文档库，解决 AI 失忆问题。
> 
> 我发现 AI 虽然不会忘记 FastAPI 是什么，但会忘记**你是怎么实现某个功能的**。比如我用 LangGraph 实现了一个复杂的多 Agent 工作流，下次开发类似功能时，AI 完全不记得我之前的实现方式，又要从头开始。
> 
> 更糟糕的是，AI 会忘记你踩过的坑。比如我在部署时遇到过 LangGraph 版本兼容问题，花了 2 小时才解决。但下次遇到类似问题，AI 又会让我走一遍弯路。
> 
> 所以我建立了一个结构化的文档库。这个结构很重要，我把文档分为：
> 
> - **已实现功能**：记录每个功能是怎么实现的，用了什么技术方案，为什么这样设计
> - **问题排查**：记录每次踩坑的经历，问题现象、根本原因、解决方案
> - **技术文档**：记录架构设计、数据模型、API 接口
> 
> 关键是**及时记录**。每次实现一个功能，立即写到'已实现功能'。每次踩坑，立即写到'问题排查'。不要等到项目结束才补文档，那时候你已经忘了细节。
> 
> 现在当我开始新功能时，我会告诉 AI：'先看 docs/后端/已实现功能/ 目录，了解我们是怎么实现类似功能的'。AI 会自动学习我的实现模式，生成的代码风格一致，不会重复踩坑。
> 
> 文档越多，AI 越聪明。这就像给 AI 建立了项目的'长期记忆'。"

## 幻灯片 4: 技巧 2 - 文档驱动的任务分解

### 标题
**解决上下文过长：文档驱动 + 模式切换**

### 描述
**问题**：
- 开发大组件时，对话超过 50 条消息
- AI 开始出现幻觉，重复犯错
- 生成的代码前后矛盾

**我的做法**：
```
1. 创建结构化的规划文档
   project-docs/dataview-refactor/
   ├── requirements.md    # 需求文档
   ├── design.md          # 设计文档
   └── tasks.md           # 任务清单

2. 利用 Costrict 的模式系统
   - 编辑 .md 文件时，自动进入 🏗️ Architect 模式
   - 编辑代码时，自动进入 💻 Code 模式
   - 复杂任务时，自动进入 📋 Orchestrator 模式

3. 任务分解策略
   - 大组件分解为小任务（每个任务 < 20 条对话）
   - 每个任务完成后立即文档化
   - 新窗口开始新任务时，让 Costrict 读取文档快速上下文
```

**Costrict 的智能模式系统**：
- 🏗️ **Architect 模式**：编辑 .md 文件时触发，强制设计优先
- 💻 **Code 模式**：编辑代码时触发，专注实现
- 📋 **Orchestrator 模式**：多步骤任务时触发，协调执行

**实际案例**：
- DataView 重构：30+ 个小任务
- 每个任务在独立窗口完成（避免上下文过长）
- 通过文档保持一致性

### 演讲稿
> "第二个技巧是文档驱动的任务分解，解决上下文过长问题。
>
> 我在重构 DataView 页面时遇到了这个问题。这个页面很复杂，包含多个数据可视化组件、实时更新、状态管理等。我和 Costrict 对话了 60 多条消息后，它开始出现幻觉：
> - 重复犯同样的错误
> - 生成的代码前后矛盾
> - 忘记之前的设计决策
>
> 后来我改用**文档驱动的方式**。我先创建规划文档，让 Costrict 帮我做设计：
> ```
> project-docs/dataview-refactor/
> ├── requirements.md    # 需求：要实现哪些功能
> ├── design.md          # 设计：技术方案和架构
> └── tasks.md           # 任务：分解为 30+ 个小任务
> ```
>
> Costrict 有个很聪明的特性：当我编辑 .md 文件时，它会自动进入 🏗️ **Architect 模式**。这个模式下它只能编辑 .md 文件，不能直接改代码，**强制我先做设计**。
>
> 关键是**任务分解**。我把 DataView 重构分解为 30 多个小任务，比如：
> - 实现热度对比图表组件
> - 实现关键词云组件
> - 实现实时数据更新
> - 优化性能
>
> 每个任务我在**独立的窗口**完成，这样每次对话不会超过 20 条，不会出现幻觉。当我开始实现代码时，Costrict 自动切换到 💻 **Code 模式**，专注于代码实现。
>
> 最强大的是，即使我关闭窗口，第二天继续开发，Costrict 也能快速恢复上下文。我只需要说：'读取 project-docs/dataview-refactor/tasks.md，继续下一个未完成的任务'。Costrict 会自动分析文档，知道：
> - 项目的需求是什么
> - 技术方案是什么
> - 已经完成了哪些任务
> - 下一步要做什么
>
> 不需要我重新解释背景，不需要担心 Costrict 忘记之前的决策。
>
> 当任务很多时，Costrict 还会自动进入 📋 **Orchestrator 模式**，帮我协调多个任务的执行顺序。
>
> **BTW**，Costrict 还有个很强的能力：智能文件搜索。它支持正则表达式搜索整个项目，并且返回的不只是匹配行，还有上下文代码。我在重构时经常用它找出所有调用某个旧 API 的地方，一个命令就能定位几十处代码，比手动搜索高效太多了。
>
> 这个方法特别适合大型重构。把大任务分解为小任务，每个任务独立完成，通过文档保持一致性。"

## 幻灯片 5: 技巧 3 - 用 .cospec 规则控制 Costrict 行为

### 标题
**让 Costrict 遵循项目规范：.cospec 配置**

### 描述
**问题**：
- Costrict 生成的代码风格不一致
- 经常建议错误的技术栈
- 不遵循项目的最佳实践

**我的做法**：
```
.cospec/
├── ARCH.md                      # 架构文档
├── CONVENTIONS.md               # 编码规范
├── TEST_GUIDE.md                # 测试指导（自动识别）
└── PROJECT_RULES.md             # 项目特定规则

关键规则：
- 技术栈约束（FastAPI、Vue 3、Pinia）
- 代码风格（PEP 8、TypeScript）
- 文档格式规范
- 测试策略（编码完成后自动执行）
```

**Costrict 的模式系统**：
- 💻 Code 模式：专注代码编写
- 🏗️ Architect 模式：专注架构设计
- 🔧 Debug 模式：专注问题排查
- 📋 Orchestrator 模式：协调多步骤任务

**实际效果**：
- Costrict 自动遵循项目规范
- 编码完成后自动查找 TEST_GUIDE.md 并执行测试
- 不再建议错误的技术栈

### 演讲稿
> "第三个技巧是用 Costrict 的 .cospec 配置系统控制 AI 行为。
> 
> 我在项目根目录创建了 `.cospec/` 目录，这是 Costrict 的项目级配置目录。这些配置会被 Costrict 自动识别和加载。
> 
> 我的配置文件包括：
> 
> **ARCH.md**：架构文档
> - 系统架构：后端 FastAPI + LangGraph，前端 Vue 3 + Pinia
> - 数据流设计：7 个 Agent 节点如何协作
> - 技术决策：为什么选择这些技术
> 
> **CONVENTIONS.md**：编码规范
> - Python 代码必须遵循 PEP 8，必须加类型提示
> - Vue 3 必须用 Composition API，不要用 Options API
> - 样式必须用 Tailwind CSS，不要写自定义 CSS
> 
> **TEST_GUIDE.md**：测试指导
> - 这个文件很特殊，Costrict 会在编码完成后**自动查找并执行**
> - 定义了测试策略：单元测试用 pytest，前端测试用 Vitest
> - 定义了测试流程：写完代码 → 自动运行测试 → 修复失败的测试
> 
> **PROJECT_RULES.md**：项目特定规则
> - 文档格式：所有文档必须包含时间戳、元信息、中文命名
> - 目录结构：遵循项目的目录规范
> - MCP 工具使用：什么时候用 Context7、GitHub MCP、Firecrawl
> 
> **Costrict 的模式系统**：
> Costrict 有一个很强大的模式系统，会根据任务自动切换：
> - 💻 **Code 模式**：写代码时自动进入，专注于代码编写
> - 🏗️ **Architect 模式**：做架构设计时进入，只能编辑 .md 文件，强制你先设计后编码
> - 🔧 **Debug 模式**：排查问题时进入，专注于问题定位
> - 📋 **Orchestrator 模式**：复杂任务时进入，协调多个步骤
>
> **BTW**，Costrict 其实还有 **🔍 Review 模式**，专门用于代码审查。它能自动检测 bugs、安全漏洞、性能问题和代码异味，对于 Pull Request 审查特别有用。我有时会在提交代码前说'进入 Review 模式，检查安全问题'，它会给出非常专业的审查报告。
>
> 关键是这些配置是**自动生效**的。我不需要每次都告诉 Costrict '用 FastAPI'、'用 Pinia'。Costrict 会自动遵循 .cospec 中的规则。
> 
> 更强大的是，当我写完代码后，Costrict 会自动查找 TEST_GUIDE.md，然后按照里面定义的测试策略执行测试。如果测试失败，它会自动修复。
> 
> 比如我说'创建一个新的 API 路由'，Costrict 会自动：
> - 进入 Code 模式
> - 读取 CONVENTIONS.md，使用 FastAPI 的最新写法
> - 添加类型提示，遵循 PEP 8
> - 生成 OpenAPI 文档
> - 写完后自动查找 TEST_GUIDE.md 并运行测试
> 
> 整个流程一气呵成，不需要我提醒。"

## 幻灯片 6: 技巧 4 - 用 MCP 工具增强 AI 能力

### 标题
**让 AI 更靠谱：MCP 工具组合拳**

### 描述
**问题**：
- AI 的记忆不可靠，经常用过时的 API
- 部署时版本对齐问题很难解决
- 不知道开源项目是怎么实现的

**我使用的 MCP 工具**：
```
1. Context7 - 查询最新官方文档
   - FastAPI 最新 API
   - LangGraph 最新用法
   - Vue 3 最新特性
   
2. GitHub MCP - 搜索开源代码
   - 搜索类似实现
   - 学习最佳实践
   - 解决版本兼容问题
   
3. Firecrawl - 抓取网页内容
   - 抓取技术文章
   - 提取配置示例
```

**关键原则**：
- ✅ 合理使用 MCP 搜索代码比让 LLM 用记忆写代码更牢靠
- ✅ 特别是部署时，版本对齐问题用这个方法特别好解决

### 演讲稿
> "第四个技巧是用 MCP 工具增强 Costrict 的能力。
> 
> 我发现 Costrict 的记忆不可靠。比如 LangGraph 最近更新了 API，但 Costrict 还在用旧版本的写法。我告诉它'用最新版本'，它说'好的'，但生成的代码还是旧的。
> 
> 后来我开始使用 MCP 工具。我配置了三个工具：
> 
> **Context7**：查询最新官方文档
> 当我需要实现某个功能时，我会说：'用 Context7 查询 LangGraph 的最新 StateGraph 用法'。Costrict 会自动调用 Context7，获取最新的官方文档，然后基于最新文档生成代码。
> 
> 这比让 Costrict 用记忆写代码靠谱多了。因为文档是实时的，不会过时。
> 
> **GitHub MCP**：搜索开源代码
> 当我遇到复杂问题时，我会说：'在 GitHub 上搜索 LangGraph 多 Agent 协作的实现'。Costrict 会搜索相关的开源项目，学习别人是怎么实现的。
> 
> 这特别适合解决版本兼容问题。比如我在部署时遇到 LangGraph 和 LangChain 版本冲突，我让 Costrict 搜索 GitHub 上的 requirements.txt，看别人用的是什么版本组合。很快就找到了可用的版本。
> 
> **Firecrawl**：抓取网页内容
> 当我需要参考技术文章或配置示例时，我会说：'抓取这篇文章的内容'。Costrict 会提取文章的关键信息，然后应用到我的项目中。
> 
> 关键原则是：**合理使用 MCP 搜索代码比让 LLM 用记忆写代码更牢靠**。特别是在部署时，版本对齐问题用这个方法特别好解决。
> 
> 现在我的工作流是：
> 1. 用 Context7 查询最新文档
> 2. 用 GitHub MCP 搜索开源实现
> 3. 结合两者生成代码
> 
> 这样生成的代码既符合最新标准，又借鉴了最佳实践。
>
> **BTW**，Costrict 还能控制浏览器进行 E2E 测试。我有时会让它'启动浏览器，测试用户注册流程'，它会自动打开浏览器、填写表单、提交，然后截图验证结果。这对于测试前端功能特别方便，省去了手动点击的时间。"

## 幻灯片 7: 技巧 5 - 立即回滚避免深陷

### 标题
**AI 犯错时的黄金法则：立即回滚**

### 描述
**问题**：
- AI 犯一个错误后，改来改去越改越乱
- 不知道该回滚到哪个版本
- 浪费大量时间在错误的方向上

**我的做法**：
```
1. 每次重大修改前创建 Git 提交
   git add .
   git commit -m "feat: 实现热点采集功能"
   
2. AI 犯错时立即回滚
   git reset --hard HEAD
   或者
   git checkout <commit-hash>
   
3. 换一个方向重新开始
   - 不要在错误的方向上继续
   - 重新思考问题
   - 尝试不同的方法
```

**关键原则**：
- ✅ 发现 AI 连续犯 2 次同样的错误 → 立即回滚
- ✅ 发现代码越改越乱 → 立即回滚
- ✅ 不要舍不得已经写的代码 → 果断回滚

### 演讲稿
> "第五个技巧是 Costrict 犯错时立即回滚，避免深陷泥潭。
> 
> 我遇到过这种情况：Costrict 在实现一个功能时犯了一个小错误，我让它修复。它改了，但引入了新的错误。我再让它修复，它又引入了更多错误。就这样改来改去，代码越来越乱，最后完全不能用了。
> 
> 我浪费了 2 个小时，最后不得不推倒重来。
> 
> 后来我学会了**立即回滚**。我的规则是：
> 
> **每次重大修改前创建 Git 提交**
> 比如我要实现一个新功能，我会先提交当前代码：
> ```bash
> git add .
> git commit -m "feat: 开始实现热点采集功能"
> ```
> 
> 这样如果 Costrict 犯错，我可以立即回滚到这个干净的状态。
> 
> **发现 Costrict 连续犯 2 次同样的错误 → 立即回滚**
> 如果 Costrict 修复一个错误后，又犯了同样的错误，说明它陷入了循环。这时候不要继续，立即回滚：
> ```bash
> git reset --hard HEAD
> ```
> 
> 然后换一个方向重新开始。比如换一个技术方案，或者把任务分解得更小。
> 
> **发现代码越改越乱 → 立即回滚**
> 如果你感觉代码越改越乱，文件越来越多，逻辑越来越复杂，说明方向错了。不要舍不得已经写的代码，果断回滚。
> 
> 关键是**不要在错误的方向上浪费时间**。回滚不是失败，而是及时止损。
> 
> 现在我的工作流是：
> 1. 重大修改前先提交
> 2. Costrict 犯错立即回滚
> 3. 重新思考问题
> 4. 换一个方向尝试
> 
> 这样我避免了很多无谓的浪费。有时候回滚重来反而更快。"

## 幻灯片 8: 技巧 6 - 让 AI 教我理解代码

### 标题
**从依赖到理解：让 AI 成为老师**

### 描述
**问题**：
- 只会用 AI 生成代码，不理解原理
- 担心变成"代码搬运工"
- 遇到问题不知道如何调试

**我的做法**：
```
1. 每次生成代码后，问 AI
   "这段代码是如何工作的？"
   "为什么要用 StateGraph 而不是 Graph？"
   "这个设计有什么优缺点？"
   
2. 让 AI 生成架构文档
   "解释整个 LangGraph 工作流的数据流动"
   "画出 Vue 组件之间的通信关系"
   
3. 遇到问题时深入理解
   "为什么会出现这个错误？"
   "根本原因是什么？"
   "如何避免类似问题？"
```

**实际效果**：
- 理解了 LangGraph 的核心概念
- 学会了 Vue 3 的最佳实践
- 能够独立调试和优化代码

### 演讲稿
> "第六个技巧是让 Costrict 教我理解代码，从依赖到理解。
> 
> 一开始我很担心：如果总是让 Costrict 写代码，我会不会变成'代码搬运工'，什么都不懂？
> 
> 后来我改变了使用方式。我把 Costrict 当成老师，而不是代码生成器。
> 
> **每次生成代码后，我会问 Costrict**：
> - '这段代码是如何工作的？'
> - '为什么要用 StateGraph 而不是普通的 Graph？'
> - '为什么要用 ref 而不是 reactive？'
> - '这个设计有什么优缺点？'
> 
> Costrict 会给出详细的解释。比如它告诉我：
> - StateGraph 适合需要在节点间传递状态的场景，而普通 Graph 不支持状态管理
> - ref 适合基本类型（数字、字符串），reactive 适合对象，因为响应式原理不同
> - 这个设计的优点是解耦，缺点是增加了一层抽象，可能影响性能
> 
> **对于整个项目，我让 Costrict 生成架构文档**：
> - 7 个 Agent 节点如何协作
> - State 数据如何在节点间流动
> - 前端组件如何通过 Pinia 共享状态
> - 为什么采用这种架构
> 
> 这不是简单的文件列表，而是深入的技术解释。
> 
> **遇到问题时，我会深入理解**：
> 比如部署时遇到 LangGraph 版本兼容问题，我不是让 Costrict 直接修复，而是问：
> - '为什么会出现这个错误？'
> - '根本原因是什么？'
> - '如何避免类似问题？'
> 
> Costrict 会解释版本依赖关系、API 变更历史、最佳实践。这样我下次就不会再犯同样的错误。
> 
> 通过这种方式，我不仅完成了项目，还深入理解了 LangGraph 和 Vue 3。现在我可以独立调试和优化代码，不再完全依赖 Costrict。
> 
> 记住：AI 时代，理解代码的工作原理比会写代码更重要。"
