## 🎯 工作流程规范

### 开始新任务前

1. ✅ 查看对应的需求清单
   - 前端: `docs/前端/需求清单_YYYY-MM-DD.md`
   - 后端: `docs/后端/需求清单_YYYY-MM-DD.md`

2. ✅ 检查API接口文档
   - 确认接口定义
   - 检查实现状态

3. ✅ 查看已实现功能
   - 避免重复开发
   - 了解现有实现

### 完成任务后

1. ✅ **优先更新现有文档**
   - 检查是否有相关的现有文档
   - 在现有文档中添加新内容
   - 避免创建重复或冗余的文档

2. ✅ 更新已实现功能文档
   - 记录实现时间
   - 记录技术要点
   - 标记遗留问题

3. ✅ 更新需求清单
   - 标记状态为已完成
   - 更新依赖关系

4. ✅ 创建技术文档（仅在必要时）
   - 复杂功能需要详细技术文档
   - 使用技术文档模板
   - 确认没有现有文档可以扩展

5. ✅ 记录问题（如遇到）
   - 使用问题排查模板
   - 特别标注死循环问题

### 测试过程中

1. ✅ **发现问题立即记录到测试问题清单**
   - 前端问题 → `docs/前端/测试问题清单_YYYY-MM-DD.md`
   - 后端问题 → `docs/后端/测试问题清单_YYYY-MM-DD.md`
   - 记录问题描述、错误日志、根本原因
   - 标记优先级和状态

2. ✅ **修复问题后更新状态**
   - 更新问题状态为"已修复"
   - 记录解决方案和修复代码
   - 验证修复是否有效

3. ✅ **定期回顾测试问题清单**
   - 检查是否有遗漏的问题
   - 更新问题统计
   - 总结常见问题模式

## 🏗️ 技术架构要点

### 系统环境

**重要**: 项目运行在 Windows PowerShell 环境
- ✅ Shell: PowerShell (cmd)
- ✅ 操作系统: Windows
- ✅ 命令语法: 使用 PowerShell 命令，不是 bash
- ❌ 不要使用 bash 语法（如 `curl -H`）

### PowerShell 常用命令

```powershell
# HTTP 请求
Invoke-WebRequest -Uri "http://localhost:8000/api" -Method POST -Body '{"key":"value"}' -ContentType "application/json"

# 或使用简写
Invoke-RestMethod -Uri "http://localhost:8000/api" -Method POST -Body '{"key":"value"}' -ContentType "application/json"

# 文件操作
Get-Content file.txt
Test-Path file.txt
Copy-Item source.txt destination.txt

# 进程管理
Get-Process | Where-Object {$_.Name -like "*python*"}
Stop-Process -Name python
```

## 📞 遇到问题时

### ⚠️ 重要：先查看历史问题！

1. **必须先查看** `docs/前端/问题排查/` 或 `docs/后端/问题排查/`
   - 查找是否有类似问题
   - 查看之前的解决方案
   - **避免重复犯错**

2. 如果是新问题，**立即创建问题排查文档**
   - 使用问题排查模板
   - 详细记录问题现象
   - 记录解决步骤
   - 标注容易进入死循环的问题
   - **不要等到解决后才记录，发现问题立即记录！**

3. 如果是重复问题
   - 在原问题文档中添加"再次发生"记录
   - 分析为什么会再次发生
   - 更新预防措施

### 🔴 已知常见问题（必读）

1. **数据库配置问题**
   - 数据库配置在 `.env` 文件，不在 `config.yaml`
   - 迁移脚本使用同步驱动（psycopg2），需要将 `postgresql+asyncpg://` 转换为 `postgresql://`
   - 参考: [问题_数据库迁移脚本编码错误](../../docs/后端/问题排查/问题_数据库迁移脚本编码错误_2026-01-17.md)

2. **文档位置问题**
   - 需求清单放在二级目录 `docs/前端/需求清单/` 或 `docs/后端/需求清单/`
   - 已实现功能、技术文档、测试文档、问题排查都在二级目录
   - 参考: [问题_文档位置错误](../../docs/后端/问题排查/问题_文档位置错误_2026-01-17.md)

3. **异步数据库查询问题**
   - FastAPI使用 `AsyncSession`，不能使用同步的 `.query()` 方法
   - 必须使用 `await db.execute(select(...))` 和 `.scalars().all()`
   - 所有异步函数调用都要加 `await`（包括Redis客户端）
   - API服务运行在端口 **8888**，不是 8000
   - 参考: [问题_预设API异步数据库查询错误](../../docs/后端/问题排查/问题_预设API异步数据库查询错误_2026-01-17.md)

4. **API端口冲突问题**
   - 重启API时可能有多个进程同时监听同一端口
   - 导致请求超时、连接失败
   - 必须先检查端口占用 (`netstat -ano | Select-String "8888"`)
   - 强制停止所有占用进程后再启动
   - 参考: [问题_API端口冲突导致超时](../../docs/后端/问题排查/问题_API端口冲突导致超时_2026-01-17.md)

5. **API测试工具选择错误** 🆕 ⚠️ 极其重要！
   - **PowerShell 的 `Invoke-RestMethod` 与 FastAPI 不兼容**
   - 会导致超时，但API实际正常工作
   - **永远优先使用 Python 测试脚本**，而不是 PowerShell 命令
   - 如果 `Invoke-RestMethod` 失败，立即尝试 `curl` 或 Python
   - **不要在同一个工具上反复尝试，浪费额度**
   - 测试工具优先级: Python requests > curl > Invoke-RestMethod (避免)
   - 参考: [问题_API测试工具选择错误导致死循环](../../docs/后端/问题排查/问题_API测试工具选择错误导致死循环_2026-01-18.md)

## 🎯 Kiro AI 自我提醒

### 文档管理检查清单

在创建新文档前，问自己：
1. ❓ 是否有相关的现有文档可以更新？
2. ❓ 这个内容是否可以作为章节添加到现有文档？
3. ❓ 创建新文档是否真的必要？
4. ❓ 新文档是否会导致信息分散或重复？

**原则**: 宁可在现有文档中添加内容，也不要轻易创建新文档。

### 每次遇到错误时的检查清单

1. ⏸️ **暂停** - 不要急于修复
2. 🔍 **检查** - 这个问题是否在问题排查文档中？
3. 🔄 **换方法** - 如果同一个方法失败2次，立即尝试其他方法
4. 📝 **记录** - 如果是新问题，立即创建问题排查文档
5. 🧠 **分析** - 为什么会犯这个错误？根本原因是什么？
6. 🔧 **修复** - 全面修复，不要只改一处
7. ✅ **验证** - 确保修复有效
8. 📚 **更新** - 更新问题排查文档和项目指南

### 防止重复犯错的原则

- ❌ **不要**: 遇到错误就立即修改代码
- ✅ **应该**: 先查看是否有类似问题的记录
- ❌ **不要**: 只修复表面问题
- ✅ **应该**: 分析根本原因，全面修复
- ❌ **不要**: 等到解决后才记录
- ✅ **应该**: 发现问题立即记录，边解决边更新
- ❌ **不要**: 连续犯同类错误
- ✅ **应该**: 第二次犯错时立即停下来反思
- ❌ **不要**: 在同一个失败的方法上反复尝试
- ✅ **应该**: 失败2次后立即切换方法

### 防止浪费额度的规则 🆕

**如果同一个操作失败3次，必须：**
1. ⏸️ 停止当前操作
2. 🤔 分析是否在错误的方向上
3. 🔄 尝试完全不同的方法
4. 📝 记录问题到文档

**测试完成标准** 🆕 ⚠️ 极其重要！
- ❌ **不要**: 在测试未全部通过前声称任务完成
- ✅ **应该**: 只有当测试结果显示 "X/X 通过" 且 X=总数时，才能说完成
- ❌ **不要**: 忽略失败的测试用例
- ✅ **应该**: 修复所有失败的测试，直到 100% 通过
- ❌ **不要**: 用"基本完成"、"大部分完成"等模糊说法
- ✅ **应该**: 明确标注 "4/6 通过，2个失败需修复"

**示例**:
```
错误: "✅ 所有测试通过！" (实际是 4/6)
正确: "⚠️ 4/6 通过，2个测试失败需修复"
```

**特别注意 - API测试：**
- ✅ **优先使用 Python 测试脚本**
- ✅ 如果 PowerShell 命令失败，立即尝试 `curl` 或 Python
- ❌ 不要反复使用 `Invoke-RestMethod`
- ❌ 不要在没有新信息的情况下反复重启API
- ❌ 不要添加大量日志但不改变测试方法
